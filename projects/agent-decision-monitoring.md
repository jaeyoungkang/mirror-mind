# 에이전트 의사결정 모니터링

> LLM 에이전트가 좋은 동료인지 검증하는 방법론

---

## 문제 정의

시스템 모니터링(서버 상태, 응답 시간)은 "서비스가 작동하는가"를 본다. 우리가 필요한 건 다르다:

- 적절한 연구 방법론을 선택했는가
- 사용자와 대화가 자연스러웠는가
- 적절한 도구 호출이 되었는가
- LLM의 의사결정이 올바랐는가
- 프로그래머틱 도구 호출(코드 레벨 오케스트레이션)이 적절했는가

즉 **"에이전트가 좋은 동료로서 행동했는가"**를 검증하는 것이다.

---

## 평가 구조 — 4개 계층

평가는 4개 계층으로 구성된다. 일반론(어떤 에이전트든) → 서비스 특화 → 도구별 → 대화 단계별.

### 1. 도구 선택·실행 (Tool Decision)

에이전트가 올바른 도구를 올바른 순서로 호출했는가.

| 차원 | 설명 | 예시 |
|------|------|------|
| 선택 정확성 | 맥락에 맞는 도구를 골랐는가 | "새 분야 진입"에 search → overview가 맞는데, 바로 analyze를 호출함 |
| 순서 적절성 | 도구 호출 순서가 논리적인가 | 검색 없이 종합 분석 시도 |
| 파라미터 품질 | 도구에 전달한 인자가 적절한가 | 검색 쿼리가 너무 넓거나 좁음 |
| 불필요 호출 | 불필요한 도구 호출이 있었는가 | 이미 충분한 데이터가 있는데 재검색 |
| 누락 | 호출했어야 할 도구를 안 했는가 | 분석 후 사용자에게 설명 없이 종료 |

**참고 프레임워크:** BFCL V3/V4 (multi-turn tool calling), ToolSandbox (stateful evaluation), T-Eval (step-level tool prediction)

### 2. 대화 품질 (Conversation Quality)

에이전트가 자연스럽고 유용한 대화를 했는가.

| 차원 | 설명 | 예시 |
|------|------|------|
| 맥락 유지 | 이전 대화를 기억하고 이어가는가 | 방금 논의한 논문을 모르는 척 |
| 탐구성 | 사용자의 의도를 적극적으로 탐색하는가 | "어떤 분야를 연구하시나요?"만 반복 |
| 정보성 | 유용한 정보를 적절히 제공하는가 | 문서만 만들고 설명 없이 끝남 |
| 개입 타이밍 | 적절한 시점에 제안/질문하는가 | 사용자가 방향을 못 잡는데 기다리기만 |
| 투명성 | 자신의 행동과 판단 근거를 설명하는가 | 왜 이 도구를 선택했는지 설명 없음 |
| 자연스러움 | 동료 연구자처럼 대화하는가 | 정해진 대사 반복, 기계적 응답 |

**참고 프레임워크:** FED 11차원 (Understanding, Informativeness, Inquisitiveness), MINT (multi-turn feedback)

### 3. 협업 적절성 (Collaboration Quality)

에이전트가 동료로서 적절한 역할을 했는가.

| 차원 | 설명 | 예시 |
|------|------|------|
| 지원 vs 대체 | 사용자 의사결정을 지원했는가, 대체했는가 | 사용자에게 묻지 않고 핵심 논문 선정→분석까지 자동 수행 |
| 확신도 전달 | 불확실할 때 솔직히 표현하는가 | 초록만 읽고 확정적 분석 제공 |
| 주도권 균형 | 사용자-에이전트 간 주도권이 적절한가 | 에이전트가 모든 것을 결정하거나, 반대로 모든 것을 물어봄 |
| 진행도 | 목표를 향해 실질적으로 나아갔는가 | 검색만 반복하고 깊이 있는 탐구 없음 |

**참고 프레임워크:** AgentBoard Progress Rate, Human-AI Collaborative Decision Making 3모드(AI-Centric / Human-Centric / Symbiotic), Trust Calibration

### 4. Lighthouse 특화 (Service-Specific)

서비스 원칙에서 도출된 평가 차원.

| 차원 | 근거 원칙 | 평가 질문 |
|------|----------|----------|
| 연구 의도 파악 | 동료원칙 1 | 쿼리 너머의 연구 의도를 파악했는가? 탐색적 vs 구체적을 구분했는가? |
| 역할 분담 제안 | 동료원칙 2 | "이건 제가, 이 판단은 님이"라는 분담이 있었는가? |
| 자율성 수준 | 동료원칙 4, L1~L5 | 맥락에 맞는 자율성 레벨을 선택했는가? |
| 시나리오 인지 | 시나리오 1~5 | 현재 어떤 연구 시나리오인지 파악하고 그에 맞게 행동했는가? |
| 톤 | 톤 규정 | 동료 톤인가, 서비스 톤으로 빠졌는가? |
| 문서 품질 | 연구 문서 패러다임 | 생성된 문서가 시나리오에 맞는 구조인가? 데이터 나열이 아니라 연구 브리프인가? |

### 5. 도구별 사용 품질 (Tool-Level)

각 도구를 "잘 사용했는가"를 평가한다.

| 도구 | 잘 쓴 기준 | 못 쓴 예시 |
|------|----------|----------|
| search_papers | 쿼리가 연구 의도에 맞는가? 키워드 조합이 적절한가? 실패 시 전략을 바꿨는가? | 같은 패턴의 쿼리 반복, 너무 넓거나 좁은 쿼리 |
| analyze_papers | 분석 대상 선정이 적절한가? 핵심 논문을 골랐는가? | 중요하지 않은 논문을 분석, 초록만으로 확정적 결론 |
| propose | 제안 타이밍과 내용이 적절한가? 근거가 있는가? | 의미 없는 제안, 맥락 무시한 제안 |

### 6. 대화 단계별 품질 (Phase-Level)

세션의 구조적 품질을 평가한다. 단계는 의존 관계가 있다 — **앞 단계가 잘 되어야 뒤 단계가 의미 있다.**

```
관계 형성 → 의도 파악 → 확산 → 수렴 → 심화 → 점검 → 결론
   ↓           ↓
  이후 행동의    이후 도구 호출의
  인식 결정      적절성 기준 결정
```

| 단계 | 목적 | 평가 기준 |
|------|------|----------|
| **관계 형성** | 첫 만남, 신뢰 구축 | 따뜻함이 먼저인가? 자기소개가 자연스러운가? 이후 행동의 인식을 결정한다 |
| **의도 파악** | 사용자가 뭘 원하는지 이해 | 충분히 물어봤는가? 표면 쿼리 너머를 파악했는가? 이후 도구 호출의 적절성 기준을 결정한다 |
| **확산** | 다양한 방향 탐색 | 검색 커버리지, 키워드 다양성, 분야 지형 파악 |
| **수렴** | 핵심 선별, 방향 좁히기 | 선별 기준이 있는가? 근거를 제시했는가? 사용자와 합의했는가? |
| **심화** | 선별된 대상 깊이 분석 | 분석 깊이, 인사이트 품질, 맥락 연결 |
| **중간 점검** | 여기까지 정리 | 적절한 타이밍인가? 요약이 정확한가? |
| **결론/전환** | 매듭, 다음 방향 | 실질적으로 나아갔는가? 다음 단계가 명확한가? |

**핵심 원칙: 의도 파악 단계 점수가 낮으면, 이후 도구 사용 평가의 맥락이 달라진다.** 의도를 잘못 파악한 상태에서 "검색 쿼리가 적절했는가"를 평가하는 것은 무의미하다. 평가자(LLM-as-Judge)는 이 의존 관계를 인지해야 한다.

## 평가 방법

### A. LLM-as-Judge (자동)

대화 로그를 다른 LLM이 평가한다.

**구조:**
1. 대화 전체 transcript (사용자 입력 + 에이전트 응답 + 도구 호출 기록)를 입력
2. 다차원 루브릭에 따라 각 차원별 0-3 점수 + 근거
3. 편향 완화: 위치 무작위화, 구조화된 루브릭, few-shot 예시

**루브릭 예시 (도구 선택 정확성):**
- 3: 맥락에 완벽히 맞는 도구를 선택, 파라미터도 적절
- 2: 대체로 맞지만 더 나은 선택지가 있었음
- 1: 부분적으로 맞지만 핵심을 놓침
- 0: 맥락에 맞지 않는 도구 선택

**참고:** MT-Bench, Agent-as-Judge 패러다임. 단일 LLM이 아닌 에이전트가 행동의 전체 체인을 평가하는 방식이 2024-2025년 부상.

### B. 골든 시나리오 비교 (반자동)

사전에 정의한 "이상적 흐름"과 실제 흐름을 비교한다.

**구조:**
1. 시나리오별 기대 행동 시퀀스 정의 (예: "새 분야 진입" → 검색 → 핵심 선별 → 분석 → 개관 → 대화)
2. 실제 에이전트 trajectory 기록
3. Progress Rate로 기대 대비 달성도 측정
4. 이탈 지점(deviation point) 자동 식별

**참고:** AgentBoard, ToolSandbox Milestones/Minefields

### C. 세션 리뷰 (수동)

실제 사용 세션을 사람이 직접 리뷰한다.

**구조:**
1. 대화 로그 + 도구 호출 타임라인을 재생 가능한 형태로 제공
2. 리뷰어가 각 의사결정 지점에서 "적절했는가" 판단
3. 패턴 발견 → 루브릭 개선 → LLM-as-Judge에 반영

## 데이터 수집 (관측 인프라)

평가를 하려면 먼저 데이터가 필요하다.

### 필수 기록 항목

| 항목 | 설명 | 현재 상태 |
|------|------|----------|
| 대화 transcript | 사용자 입력 + 에이전트 응답 전문 | O (messages 테이블) |
| 도구 호출 로그 | 도구명, 입력, 출력, 소요시간, 성공/실패 | △ (인메모리만, DB 미영속) |
| 의사결정 근거 | LLM이 왜 이 도구를 선택했는지 (thinking) | X |
| 프로그래머틱 분기 | 코드 레벨 오케스트레이션 결정 (phase 전환, post-action 등) | △ (phase 기록은 있으나 미영속) |
| 사용자 행동 | 문서 열기/닫기, 제안 수락/거절, 검색 실행 등 | X |

### 영속화 방안

현재 인메모리 링 버퍼(20턴) → DB 테이블로 전환 필요.

```
turn_logs 테이블
  id, conversation_id, turn_index,
  phases (jsonb),      -- 파이프라인 단계별 기록
  tool_calls (jsonb),  -- 도구 호출 상세
  duration_ms,
  created_at

user_events 테이블
  id, conversation_id, event_type, payload (jsonb), created_at
```

### 산업 도구 참고

- **Langfuse** (오픈소스, self-host) — 트레이싱 + LLM-as-judge 통합. 가장 실용적
- **Braintrust** — evaluation-first 철학, 프로덕션 실패를 평가 데이터로 자동 변환
- **Arize Phoenix** — 엔터프라이즈급, 에이전트 평가 전용 기능

## 적용 순서

### Phase 1: 데이터 확보
- 도구 호출 로그 DB 영속화
- 사용자 행동 이벤트 수집
- 기존 dev 타임라인 뷰와 통합

### Phase 2: 골든 시나리오 + 수동 리뷰
- 시나리오 1(새 분야 진입)의 이상적 흐름 정의
- 실제 세션과 비교하여 이탈 지점 식별
- 패턴을 루브릭으로 정리

### Phase 3: LLM-as-Judge 자동화
- Phase 2에서 축적한 루브릭으로 자동 평가 구축
- 세션 종료 시 자동 스코어링
- 점수 추세 모니터링

### Phase 4: 에이전트 개선 루프
- 낮은 점수 세션 → 원인 분석 → 프롬프트/도구/오케스트레이션 개선
- 개선 전후 점수 비교로 효과 측정
