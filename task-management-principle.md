# 업무 관리 원칙 (Task Management Principle)

## 관리 도구
- 로컬 폴더와 문서로 관리
- 깃은 버전 관리 용도로만 사용

## 업무 수집
- 대화 중 업무가 식별되면 AI가 선제적으로 inbox 등록을 제안
- 합의 후 `tasks/inbox.md`에 기록
- 주기적으로 inbox를 함께 검토하여 구조화 및 우선순위 결정

## 맥락 관리
- 모든 환경에서 공유해야 할 맥락은 깃 관리 문서에 기록한다
- AI 로컬 메모리는 현재 환경 정보(PC, OS 등)만 보관한다
- 다른 PC에서 작업 시 깃 문서를 통해 맥락을 복원한다

## 업무 구조화
- 업무는 도메인(성격)과 프로젝트(목표 단위) 두 축으로 분류한다
- 초기에는 `tasks/projects.md`에서 플랫하게 관리한다
- 복잡도가 높아지면 프로젝트별 폴더 구조로 전환한다

## 운영 원칙
- 불확실성이 클 때는 단순한 구조로 시작하고, 실제 필요에 따라 점진적으로 개선한다
- 문서 분리/통합 판단 기준: (1) 독립적으로 의미를 갖는가 (2) 추상화로 상위 문서에 흡수 가능한가. 둘 다 아니면 통합이 기본값이다.
- 새로운 내용을 추가하기 전에 비판적 검토를 거친다. 검토 과정에서 추상화가 일어나고, 추상화가 문서 구조를 단순하게 만든다.

## 진행 관리
- 상태 표기: `[ ]` 대기 / `[>]` 진행 중 / `[x]` 완료 / `[!]` 보류 (사유 표기)
- 하위 업무는 들여쓰기로 계층 표현
- 예시:
  ```
  - [>] 상위 업무
    - [x] 하위 업무 1
    - [>] 하위 업무 2
    - [ ] 하위 업무 3
  ```

## 대화 관리
- 각 세션(비서실장/하위 프로젝트)은 대화 원시 데이터를 자기 저장소에 저장한다
- 비서실장 세션은 로컬 경로로 하위 프로젝트의 대화 기록에 접근하여 의사결정을 수렴한다
- 주요 의사결정은 `tasks/decisions.md`에 인덱싱한다
- AI가 세션 마무리 시 기록할 결정이 있는지 선제적으로 제안한다

## 기록과 회고
- 완료된 프로젝트나 마일스톤 단위로 회고를 진행한다
- 회고 시점은 AI가 선제적으로 제안할 수 있다
- 회고 결과 중 반복 적용할 교훈은 해당 원칙 문서에 반영한다
